# AGENTS.md

This repository builds an open, reproducible perp execution-quality benchmark. It collects public order books, computes execution-quality metrics, stores samples in SQLite, and exports a static public site.

## Project Goal

The benchmark exists to test whether traders trust the data and methodology enough to cite, share, dispute, and improve it. Optimize for credibility, auditability, and distribution. Do not turn this into a trading terminal or a signal product unless the milestone explicitly says so.

## Current Milestone

Read `docs/milestones/2026-06-trust-distribution-v1.md` before changing product behavior, public presentation, export shape, daily summaries, anomalies, or deployment behavior.

## Non-Goals

Do not add:

- paid access, login, custom user alerts, or per-user settings
- alpha calls, trading signals, liquidation/whale/vault dashboards, or position recommendations
- venue marketing copy or special-case ranking treatment
- raw private feeds, account-based APIs, UI scraping, or manually edited benchmark numbers
- broad dashboard/product rewrites when a milestone calls for narrow trust/distribution work

## Source Of Truth

- `src/config/markets.ts` defines venues, reference venues, markets, symbols, and listing status.
- Exchange adapters live under `src/exchanges/`.
- Metric logic lives in `src/metrics/orderbook.ts`.
- SQLite storage and query behavior live in `src/storage/sqlite.ts`.
- Static public artifacts are generated by `src/export/static.ts`.
- Checked-in `public/` files are generated artifacts. When export behavior or presentation changes, update the source and rerun `npm run export`; do not patch generated HTML alone.
- Public methodology is generated from `src/export/static.ts`, even though the output file is `public/methodology.html`.

## Metric And Ranking Rules

- Primary comparable metrics are depth and estimated taker slippage.
- Spread stays visible as a top-of-book signal, but it is tick-size sensitive and should not be presented as a primary cross-venue ranking cue.
- Keep `3bp`, `5bp`, and `10bp` depth in the same metric family.
- Keep both `100,000 USD` and `1,000,000 USD` estimated taker slippage in the slippage family.
- Hyperliquid and Aster are reference venues. They may be displayed for context, but should not participate in benchmark `Best` calculations unless the methodology changes intentionally.
- StandX `SOL-USD` remains `N/A: not listed` until public StandX symbol data supports it. Do not substitute another market.

## Data Pipeline

The intended pipeline is:

1. Exchange adapters fetch public order books.
2. Pure metric functions compute spread, depth, and slippage.
3. SQLite appends samples and status rows.
4. Export jobs write `public/data/*.json`, `public/index.html`, and `public/methodology.html`.
5. Summary and anomaly jobs derive public text/events from stored metrics.

Persistent SQLite storage matters. A disposable scheduled CI run that starts from an empty checkout is not enough for a credible 7 day history.

## Commands

Install:

```bash
npm install
```

Verify deterministic behavior:

```bash
npm test
npm run typecheck
git diff --check
```

Refresh generated static artifacts:

```bash
npm run export
```

Run one live collector/export round:

```bash
npm run run:benchmark -- --once
```

Run the realtime collector/export loop:

```bash
npm run run:benchmark
```

Dry-run anomaly detection:

```bash
npm run anomaly:dry-run
```

Send exported anomaly events to Telegram, only when credentials are configured:

```bash
npm run telegram:send-anomalies
```

## Verification Expectations

For code or public-output changes, run:

```bash
npm test
npm run typecheck
git diff --check
npm run export
```

For changes that affect adapters, metrics, data freshness, daily summaries, anomalies, or public JSON shape, also run a live one-shot when network access and credentials allow:

```bash
npm run run:benchmark -- --once
npm run anomaly:dry-run
```

After export changes, inspect:

- `public/index.html`
- `public/methodology.html`
- `public/data/latest.json`
- `public/data/history-7d.json`
- `public/data/daily-summary.json`

## Implementation Style

- Keep changes narrow and aligned with the current milestone.
- Prefer existing repo patterns over new abstractions.
- Add or update tests when changing metric calculations, SQLite schema/query behavior, export shape, summary text, anomaly rules, or public rendering logic.
